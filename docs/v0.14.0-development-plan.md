## 3. Phase B: Smart Dispatcher — Multi-Phase Execution (Day 1-2)

### Problem

Current `escalation_dispatcher.py` reads only one field from route.json:
`recommended_route`. It maps route → single API call. This ignores the
committee's full intelligence:

```json
{
  "delegability": "high",              // IGNORED
  "local_preprocessing_required": true, // IGNORED
  "pro_should_execute": false,          // IGNORED
  "pro_should_adjudicate": true,        // IGNORED
  "required_artifacts": [               // IGNORED
    "file_summary", "diff", "test_log"
  ]
}
```

The dispatcher currently only does: `task → one model → done`.
It should do: `task → prep → artifacts → execute → adjudicate → accept/reject`.

### New Execution Flow

```
route.json (full committee output)
  │
  ├── delegability = ?
  │   ├── high → decompose task, run phases in parallel
  │   ├── medium → sequential phases
  │   ├── low → single Pro call
  │   └── blocked → abort
  │
  ├── local_preprocessing_required ?
  │   ├── yes → Phase 0: local model reads files, generates artifacts
  │   │         → wait for required_artifacts
  │   └── no → skip to execution
  │
  ├── recommended_route = ?
  │   ├── local_only → local model executes
  │   ├── flash_* → Flash executes
  │   └── pro_decision → Pro executes
  │
  ├── pro_should_adjudicate ?
  │   ├── yes → Pro reviews output, returns accept|reject|retry
  │   │         ├── accept → done, return result
  │   │         ├── reject → return reason, abort
  │   │         └── retry → re-execute with Pro feedback
  │   └── no → done
  │
  └── Result
```

### Implementation Changes

**File**: `tools/escalation_dispatcher.py`

```python
@dataclass
class PhaseResult:
    phase: str
    route: str
    success: bool
    output: str
    artifacts: dict       # NEW: artifacts produced
    adjudication: str     # NEW: accept|reject|retry|none
    elapsed_ms: int
    cost_cny: float

def dispatch(route_path, task, cloud_ok, max_budget):
    route = load_route(route_path)

    # Step 0: Preprocessing
    if route.get("local_preprocessing_required"):
        artifacts = _run_local_preprocessing(task, route["required_artifacts"])
        if not artifacts_complete(artifacts, route["required_artifacts"]):
            return abort("missing artifacts")

    # Step 1: Execute
    result = _execute_phase(route["recommended_route"], task, artifacts)

    # Step 2: Adjudicate (if Pro should review)
    if route.get("pro_should_adjudicate"):
        verdict = _pro_adjudicate(task, result, artifacts)
        if verdict == "reject":
            return abort(f"Pro rejected: {verdict.reason}")
        if verdict == "retry":
            result = _execute_phase(route["recommended_route"],
                                     task + "\nPro feedback: " + verdict.feedback)
            verdict = _pro_adjudicate(task, result, artifacts)  # one retry

    return result
```

### Required New Functions

1. `_run_local_preprocessing(task, required_artifacts)` — runs local model to generate file summaries, diffs, test logs. Returns dict of artifact_name → content.

2. `_pro_adjudicate(task, result, artifacts)` — sends Flash/local output to Pro for review. Pro returns structured verdict: `{decision: accept|reject|retry, reason: str, feedback: str}`.

3. `artifacts_complete(produced, required)` — checks if all required artifacts exist and are non-empty.

### Pro Writes the Plan WITH Model Assignments

**Key insight**: Pro should decide model switching at plan time, not execution time.

Currently `draft_execution_plan.py` uses heuristic templates + TieringPolicy
to assign models. Instead, **Pro itself should write the plan AND specify
which model for each phase**:

```json
// plan.json — written by Pro, validated by committee
{
  "task": "review local-llm-pipeline for security issues",
  "phases": [
    {
      "name": "understand",
      "description": "Read router, hooks, MCP server source",
      "model": "qwen3-coder:30b",
      "route": "local_only",
      "produces": ["file_summaries", "architecture_map"]
    },
    {
      "name": "analyze",
      "description": "Analyze code for security vulnerabilities",
      "model": "deepseek-v4-flash",
      "route": "flash_subagent",
      "requires": ["file_summaries"],
      "produces": ["security_findings"]
    },
    {
      "name": "adjudicate",
      "description": "Review Flash findings, confirm or reject",
      "model": "deepseek-v4-pro",
      "route": "pro_decision",
      "requires": ["security_findings"],
      "produces": ["final_report"]
    }
  ]
}
```

**Flow**:
```
User task
  → Pro (plan-only): writes plan.json with per-phase model assignments
  → Committee: reads plan.json, validates model choices, adjusts if needed
  → route.json: committee-approved model switching schedule
  → Dispatcher: executes each phase with specified model
  → /model switches at each phase boundary
```

**What changes in draft_execution_plan.py**:
- `--real-run --cloud-ok`: calls DeepSeek Pro to write the actual plan + model assignments
- Pro's prompt includes available models and their capabilities
- Pro outputs structured JSON with per-phase `model` and `route` fields
- Each phase specifies `requires` (input artifacts) and `produces` (output artifacts)

**What changes in route_committee.py**:
- Committee reads plan.json phases
- Validates Pro's model assignments (does this really need Pro? can Flash do it?)
- Adjusts if Pro over-assigned expensive models
- Outputs validated route.json with model switching schedule

**What changes in escalation_dispatcher.py**:
- Reads route.json → gets model switching schedule
- Before each phase: `/model <specified_model>`
- After each phase: saves artifacts
- Respects `requires` — won't start phase until dependencies ready

**Benefit**: The model switching plan is explicit, auditable, and correctable.
Committee catches Pro over-assigning expensive models. Dispatcher just follows orders.

### C.4 Parallel Committee Preloading

**Problem**: Current `ThreadPoolExecutor` runs models in parallel threads — but Ollama
loads models on-demand. If model A is in memory but model B needs loading,
thread A waits for B's disk I/O (~40s for 31B). Threads = parallel requests,
not parallel model loading.

### C.2 Solution: Preload + Parallel

```python
# In local_route_committee.py, before ThreadPoolExecutor:

def _preload_models():
    """Warm up both models before debate."""
    warmup_prompt = '{"ping": true}'  # minimal prompt
    import concurrent.futures
    with concurrent.futures.ThreadPoolExecutor(max_workers=2) as pool:
        pool.submit(_call_model, MODEL_A, warmup_prompt, timeout=120)
        pool.submit(_call_model, MODEL_B, warmup_prompt, timeout=120)
    # Both loaded → subsequent debate is fast

# In convene():
_preload_models()  # parallel warmup
# ... then existing debate code
```

**Expected**: 50-65s → 20-25s (loading parallelized, inference still serial).

### C.3 Alternative: Dedicated Committee Ollama Instance

If preloading doesn't help enough:

```bash
# On zero12: run a second Ollama on dedicated port
OLLAMA_HOST=0.0.0.0:11435 OLLAMA_MODELS=/opt/committee-models ollama serve &

# Preload committee models on this instance only
ollama pull gemma4:12b-unsloth  # to 11435
ollama pull qwen3.6:27b          # to 11435

# Committee code uses port 11435
# Models always in memory → instant response
```

**Tradeoff**: ~25GB extra RAM for always-warm models.
