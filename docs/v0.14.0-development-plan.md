## 2. Phase A: Model Fleet Audit & Repair (Day 1)

**Goal**: Every model in the profiles either works or is explicitly marked unavailable.

### A.1 Diagnose gemma4:31b Q8_0 — PRIORITY #1

**Current state**: 35GB Q8_0 model. API returns HTTP 200 but response body is empty.
GGUF file exists (size > 0 on disk). Model loads (Ollama show returns details).
But `api/generate` returns zero tokens.

**Hypotheses (ranked)**:
1. **Chat template mismatch** — Gemma4 uses `<start_of_turn>` model tokens.
   Ollama may be applying wrong template → model outputs nothing
2. **Stop token fires immediately** — Template or Modelfile has a stop token
   that matches the first generated token
3. **VRAM pressure** — Q8_0 needs ~35GB. If other models loaded, may OOM silently

**Diagnosis steps (non-destructive, on zero12)**:

```bash
# 1. Check VRAM
ssh zero12 nvidia-smi

# 2. Test with explicit Gemma4 chat template
curl http://193.168.2.2:11434/api/generate -d '{
  "model": "gemma4:31b",
  "prompt": "<bos><start_of_turn>user\nSay exactly: OK\n<end_of_turn>\n<start_of_turn>model\n",
  "stream": false,
  "options": {"num_predict": 10, "temperature": 0}
}'

# 3. Compare Modelfile with working unsloth variant
ollama show gemma4:31b --modelfile > /tmp/gemma_q8.modelfile
ollama show gemma4:31b-unsloth --modelfile > /tmp/gemma_q4.modelfile
diff /tmp/gemma_q8.modelfile /tmp/gemma_q4.modelfile
# Focus on: TEMPLATE, SYSTEM, PARAMETER stop

# 4. Test with raw prompt (no template)
curl http://193.168.2.2:11434/api/generate -d '{
  "model": "gemma4:31b",
  "prompt": "Say OK",
  "stream": false,
  "options": {"num_predict": 10, "temperature": 0},
  "raw": true
}'

# 5. If nothing works: check disk
ssh zero12 ls -lh /usr/share/ollama/.ollama/models/blobs/ | grep -i gemma
ssh zero12 ollama ps  # see what's loaded
```

**Fix candidates (apply only after root cause confirmed)**:
- Template fix → `ollama create gemma4:31b -f fixed.Modelfile`
- Stop token fix → remove problematic PARAMETER stop from Modelfile
- VRAM fix → unload other models first, then test
- GGUF re-download → `ollama pull gemma4:31b` (LAST RESORT)

**Success**: gemma4:31b Q8_0 returns non-empty response to "Say OK"

## 4. Phase C: Parallel Committee Preloading (Day 2-3)

### C.1 Problem

Current `ThreadPoolExecutor` runs models in parallel threads — but Ollama
loads models on-demand. If model A is in memory but model B needs loading,
thread A waits for B's disk I/O (~40s for 31B). Threads = parallel requests,
not parallel model loading.

### C.2 Solution: Preload + Parallel

```python
# In local_route_committee.py, before ThreadPoolExecutor:

def _preload_models():
    """Warm up both models before debate."""
    warmup_prompt = '{"ping": true}'  # minimal prompt
    import concurrent.futures
    with concurrent.futures.ThreadPoolExecutor(max_workers=2) as pool:
        pool.submit(_call_model, MODEL_A, warmup_prompt, timeout=120)
        pool.submit(_call_model, MODEL_B, warmup_prompt, timeout=120)
    # Both loaded → subsequent debate is fast

# In convene():
_preload_models()  # parallel warmup
# ... then existing debate code
```

**Expected**: 50-65s → 20-25s (loading parallelized, inference still serial).

### C.3 Alternative: Dedicated Committee Ollama Instance

If preloading doesn't help enough:

```bash
# On zero12: run a second Ollama on dedicated port
OLLAMA_HOST=0.0.0.0:11435 OLLAMA_MODELS=/opt/committee-models ollama serve &

# Preload committee models on this instance only
ollama pull gemma4:12b-unsloth  # to 11435
ollama pull qwen3.6:27b          # to 11435

# Committee code uses port 11435
# Models always in memory → instant response
```

**Tradeoff**: ~25GB extra RAM for always-warm models.

### C.4 Time Budget

| Scenario | Debate Time | Acceptable? |
|----------|------------|-------------|
