## 4. Phase C: Parallel Committee (Day 2-3)

### C.1 Problem

Current implementation already runs models in parallel via `ThreadPoolExecutor`:

```python
with ThreadPoolExecutor(max_workers=2) as pool:
    future_a = pool.submit(_call_model, MODEL_A, prompt)
    future_b = pool.submit(_call_model, MODEL_B, prompt)
```

But Ollama loads models on-demand. If both models aren't preloaded, one waits for the other to load from disk. Gemma 31B takes 40+ seconds to load, making Qwen wait.

### C.2 Solution: Model Preloading

Before committee convenes, send a warmup ping to both models:

```python
def _preload_models():
    """Send a tiny prompt to each model to trigger Ollama loading."""
    for model in [MODEL_A, MODEL_B]:
        _call_model(model, "ping", timeout=60)  # discard output
```

Ollama keeps recently-used models in memory. After preloading, the actual debate runs with both models already warm.

**Expected improvement**: 50s → 25s (both models loaded in parallel, then quick inference)

### C.3 Alternative: Dedicated Ollama Instances

Run two Ollama instances on different ports:
- Port 11434: main models (Qwen, Nemotron, etc.)
- Port 11435: committee models preloaded

```bash
# On zero12:
OLLAMA_HOST=0.0.0.0:11435 ollama serve &
ollama pull gemma4:12b  # preload on 11435
ollama pull qwen3.6:27b  # preload on 11435
```

Committee code points to port 11435 — both models always warm.

**Tradeoff**: More RAM usage, but consistent <20s debate time.

### C.4 Debate Time Budget

| Scenario | Time | Acceptable? |
|----------|------|-------------|
| Both preloaded, 12B+27B | 10-15s | ✓ excellent |
| Serial loading, 31B | 50-65s | ✗ too slow |
| Both preloaded, 27B+30B | 15-20s | ✓ good |
| 5070 Ti models only | 5-8s | ✓✓ best |

**Target**: <20s total debate time.
