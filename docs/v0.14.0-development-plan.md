## 2. Phase A: Model Diagnosis & Repair (Day 1)

**Principle**: Q8 precision models are more valuable than Q4 replacements.
           Diagnose root cause before any destructive action.

### A.1 Diagnose gemma4:31b (Q8_0) Empty Response

**Current state**:
| Property | gemma4:31b (standard) | gemma4:31b-unsloth |
|----------|----------------------|---------------------|
| Quantization | **Q8_0** (high precision) | Q4_0 (lower precision) |
| Parameters | 30.7B | 30.7B |
| API response | **empty** (all prompts) | OK (43s load) |
| Ollama `size` | 0 (API bug, not file size) | 0 (same API bug) |
| GGUF exists? | ✓ (model loads, just silent) | ✓ |

**Hypothesis (ranked by likelihood)**:
1. **Chat template mismatch** — Gemma4 requires `<start_of_turn>` / `<end_of_turn>` tags. Ollama may be using wrong template, causing model to output nothing
2. **Stop token misconfiguration** — Model hits stop token immediately (e.g., `<|end|>` appearing in first token)
3. **VRAM pressure on Q8** — Q8_0 requires ~31GB VRAM. If server has other models loaded, Q8 might OOM silently
4. **GGUF metadata corruption** — File is intact but metadata (tokenizer config) is wrong

**Diagnosis steps (in order, non-destructive)**:

```bash
# Step 1: Check VRAM usage on zero12
ssh zero12 nvidia-smi

# Step 2: Test with explicit Gemma chat template
curl http://193.168.2.2:11434/api/generate -d '{
  "model": "gemma4:31b",
  "prompt": "<start_of_turn>user\nSay OK\n<end_of_turn>\n<start_of_turn>model\n",
  "stream": false,
  "options": {"num_predict": 20, "temperature": 0}
}'

# Step 3: Check GGUF file on disk
ssh zero12 ls -lh /usr/share/ollama/.ollama/models/blobs/ | grep gemma

# Step 4: Compare Modelfile with unsloth variant
ollama show gemma4:31b --modelfile > /tmp/gemma_q8.modelfile
ollama show gemma4:31b-unsloth --modelfile > /tmp/gemma_q4.modelfile
diff /tmp/gemma_q8.modelfile /tmp/gemma_q4.modelfile
# Focus on: TEMPLATE, SYSTEM, PARAMETER lines

# Step 5: Test with minimal parameters
curl http://193.168.2.2:11434/api/generate -d '{
  "model": "gemma4:31b",
  "prompt": "Say the word OK and nothing else.",
  "stream": false,
  "options": {"num_predict": 10, "temperature": 0, "top_k": 1}
}'
```

**Fix candidates (apply only after diagnosis confirms root cause)**:
1. If template issue → update Modelfile with correct Gemma4 chat template
2. If stop token issue → remove problematic stop tokens from Modelfile
3. If VRAM issue → unload other models before using gemma4:31b, or use smaller quant for debate
4. If GGUF corruption → `ollama pull gemma4:31b` (last resort)

**Success criterion**: `gemma4:31b` (Q8_0) returns non-empty response to "Say OK"

## 4. Phase C: Parallel Committee (Day 2-3)

### C.1 Problem

Current implementation already runs models in parallel via `ThreadPoolExecutor`:

```python
with ThreadPoolExecutor(max_workers=2) as pool:
    future_a = pool.submit(_call_model, MODEL_A, prompt)
    future_b = pool.submit(_call_model, MODEL_B, prompt)
```

But Ollama loads models on-demand. If both models aren't preloaded, one waits for the other to load from disk. Gemma 31B takes 40+ seconds to load, making Qwen wait.

### C.2 Solution: Model Preloading

Before committee convenes, send a warmup ping to both models:

```python
def _preload_models():
    """Send a tiny prompt to each model to trigger Ollama loading."""
    for model in [MODEL_A, MODEL_B]:
        _call_model(model, "ping", timeout=60)  # discard output
```

Ollama keeps recently-used models in memory. After preloading, the actual debate runs with both models already warm.

**Expected improvement**: 50s → 25s (both models loaded in parallel, then quick inference)

### C.3 Alternative: Dedicated Ollama Instances

Run two Ollama instances on different ports:
- Port 11434: main models (Qwen, Nemotron, etc.)
- Port 11435: committee models preloaded

```bash
# On zero12:
OLLAMA_HOST=0.0.0.0:11435 ollama serve &
ollama pull gemma4:12b  # preload on 11435
ollama pull qwen3.6:27b  # preload on 11435
```

Committee code points to port 11435 — both models always warm.

**Tradeoff**: More RAM usage, but consistent <20s debate time.
