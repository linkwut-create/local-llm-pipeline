## 2. Design Decisions (Pre-Commitment)

### Decision 1: Graduated Rollout

```
Phase A: Router calibration + metrics cleanup → unblock warning gate
Phase B: Warning gate + stop hook → harden local-only path
Phase C: Flash limited real-run pilot → first real cloud calls
Phase D: Pro smoke chain + full escalation → complete chain
```

Each phase has its own stop condition: metrics must pass before next phase.

### Decision 2: Dual-Switch Safety

Real API calls require BOTH:
- `--cloud-ok` flag (user intent)
- `--real-run` flag (explicit execution)

Missing either → falls back to dry-run mock. This prevents accidental cloud calls.

### Decision 3: Budget Hard Cap

- Flash: 0.5 CNY per call, 5 CNY/day max
- Pro: 2.0 CNY per call, 10 CNY/day max
- Hard stop when budget exceeded (not advisory)

### Decision 4: API Key Handling

- `DEEPSEEK_API_KEY` env var only — never in config files, never logged
- Read immediately before call, discarded after
- Privacy gate runs BEFORE key is read

### Decision 5: What NOT to Change

Per BAN-002, BAN-003, BAN-007, BAN-008:
- No new MCP tools without explicit use case + schema design + escalation design
- No CLI output field deletion
- All new config keys must have `validate_configs` coverage
- `VERSION` only changed at release commit

### Step C.2: Implement Guarded Real-Run Adapter

**Files to modify**:
- `tools/deepseek_execution_adapter.py` — replace `real_run_not_implemented` with real API call
- `tools/cost_ledger.py` — integrate budget guard auto-block

**Design contract** (from `docs/guarded_real_run_adapter_design.md`):
```
Gate sequence:
[1] cloud_ok → required
[2] router → task must be flash-fallback or pro-review
[3] privacy → privacy_gate check (env var, not config file)
[4] budget → cost_ledger estimate ≤ budget remaining
[5] dry_run → --real-run flag required
[6] API call → POST https://api.deepseek.com/v1/chat/completions
```

**Response redaction** (mandatory):
- Strip `DEEPSEEK_API_KEY` from any error messages
- Never log response body at INFO level (DEBUG only)
- Redact thinking tokens in error reports

**Escalation triggers** (Flash pilot scope):
- Local model same task fails 2 consecutive times → escalate to Flash
- Flash output empty or invalid JSON → report failure, don't auto-escalate to Pro
- Task type must be in allowlist: `review-diff`, `summarize-file`, `summarize-tree`

**Tests**:
- `local_generate_test_plan` on `tools/deepseek_execution_adapter.py`
- New: `tests/test_flash_real_run.py` — 15+ tests covering gate sequence, redaction, retry
- Mock: all tests use mock HTTP, no real API calls in CI

**Review**: `local_debate_review_diff` (full 3-round — architecture change, touches security)

### Step D.2: Pro Escalation Logic

**Router changes**:
- Pro triggers: `risk_level == "high"` or `"critical"`, `task_type` in `{release-risk-review, security-review, architecture-review, interface-review, api-execution-boundary}`
- Flash→Pro escalation: Flash output conflicts with local → Pro arbitration

**Implementation file**: `tools/local_llm_router.py` (escalation logic)

### Step D.3: End-to-End Automatic Escalation

Wire the full chain:
```
Task received
  │
  ├─ local model available? → YES → run locally
  │   └─ FAIL 2×? → escalate to Flash
  │       └─ Flash output invalid or conflicts? → escalate to Pro
  │           └─ Pro fail? → report to controller
  │
  ├─ local model unavailable? → check cloud_ok
  │   └─ YES → route directly to Flash (if low/medium) or Pro (if high/critical)
  │   └─ NO → deferred, ask user
  │
  └─ privacy blocked? → refuse cloud, local only
```

**Integration test**: `tests/test_escalation_e2e.py` — mock HTTP, verify chain transitions
